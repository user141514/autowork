# System Overview

## 1. 系统要解决什么问题

本系统要解决的问题不是“如何让机器人回复群聊”，而是“如何把群聊中产生的开发意图，转化为可审计、可审核、可执行、可恢复的工程工作流”。

目标工作流是：

```text
聊天消息
-> 任务候选识别
-> WorkDoc 工作文档
-> 策略与风险判断
-> 人工审核或自动放行
-> Agent 执行
-> 测试
-> Git 变更发布
-> 状态回写
```

这个系统位于聊天入口、Coding Agent、测试系统和 Git 平台之间，承担“工作流控制层”的职责。它不重新实现 Coding Agent，也不把某个聊天平台当成系统核心，而是把多种外部能力组织成稳定的端到端流程。

系统需要重点解决以下问题：

- 群聊信息是非结构化、嘈杂、上下文不完整的，不能直接驱动代码修改。
- Coding Agent 擅长执行具体修改，但不天然负责组织审批、风险判断、状态恢复和审计链路。
- Git commit、branch、PR、测试结果、执行日志需要统一记录，否则执行结果不可追踪。
- 后续聊天源、Agent 执行器、Git 平台、前端界面都可能替换，核心流程不能和某个单点实现绑定。

因此，本项目的关键价值是把已有单点能力连接成一个可靠工作流，而不是重复造微信机器人、代码 Agent 或 Git 工具。

## 2. 为什么不是普通微信机器人

普通微信机器人通常围绕“收到消息后立即响应”设计，核心能力是：

- 监听聊天消息
- 解析关键词或命令
- 调用接口
- 返回文本、图片或文件

这种架构适合通知、查询、客服、轻量自动化，但不适合自主开发工作流。开发任务有更高风险和更长生命周期：

- 一条群聊消息可能只是讨论，不一定是任务。
- 一个任务可能需要多条消息共同形成上下文。
- 需求可能缺少验收标准、仓库信息、约束条件或风险说明。
- 修改代码前必须经过策略判断和必要的人工确认。
- 执行过程可能跨越分钟级甚至小时级，期间会产生日志、diff、测试结果和 Git 操作。
- 失败后需要恢复、重试、回滚或转人工，而不是简单回复“失败”。

所以聊天平台只能作为 ChatAdapter，是输入和反馈通道之一。业务核心必须放在工作流系统内部，而不是写死在微信回调、企业微信机器人或某个 IM SDK 中。

## 3. 为什么 WorkDoc 是核心契约

WorkDoc 是从聊天噪声中提炼出来的结构化任务文档，是整个系统的 Single Source of Truth。

它至少承担四个职责：

1. **语义归一**

   群聊里可能有口语、玩笑、补充说明、截图、反复确认和无关讨论。WorkDoc 把这些内容归一成工程任务需要的字段，例如问题摘要、预期行为、约束、验收标准、风险等级和证据消息。

2. **执行契约**

   AgentRunner 不应该读取原始群聊并自由理解，而应该只根据 WorkDoc 执行。这样可以保证 Agent 的输入稳定、可复现、可审计。

3. **审核载体**

   人工审核的对象不是聊天记录本身，而是 WorkDoc。审核人需要确认的是：这个任务是否清楚、是否允许执行、风险是否可接受、验收标准是否足够明确。

4. **状态锚点**

   AgentRun、GitOperation、测试结果、报告回写都应关联到 WorkDoc。这样系统可以回答：

   - 这次代码修改来自哪个任务？
   - 任务依据哪些群聊消息生成？
   - 谁批准了执行？
   - Agent 做了什么？
   - 测试是否通过？
   - commit 和 PR 在哪里？

没有 WorkDoc，系统就会退化成“聊天消息直接触发命令”。这会导致上下文不可控、责任链不清、失败难恢复。

## 4. 为什么 Agent 不能直接读群聊改代码

Agent 不能直接读取群聊并修改代码，原因不是技术上做不到，而是工程治理上不可接受。

主要风险包括：

- **意图误判**：群聊中一句“这个按钮是不是坏了”可能是讨论，不一定是授权修复。
- **上下文污染**：Agent 可能把无关讨论、玩笑、历史消息当成需求。
- **缺少验收标准**：没有 acceptance criteria 的任务无法判断是否完成。
- **权限绕过**：如果 Agent 直接吃群聊上下文，就很难在执行前做风险判断和审批。
- **不可复现**：后续无法证明 Agent 当时到底基于哪些输入做了修改。
- **审计困难**：commit 只能看到代码结果，看不到决策链路。
- **安全风险**：群聊中可能出现敏感文件、生产配置、认证、支付等高风险修改请求，必须先经过 PolicyGate。

正确方式是：

```text
ChatMessage -> TaskExtractor -> WorkDoc -> WorkDocService.validate -> PolicyGate -> AgentRunner
```

Agent 只接收已经验证、批准、风险分级后的 WorkDoc 和受控的 RepoContext。原始群聊消息只能作为证据引用，而不是 Agent 的直接执行输入。

## 5. 系统设计原则

- **聊天入口可替换**：微信、企业微信、Slack、Discord、文件导入都只是 ChatAdapter。
- **Agent 可替换**：Claude CLI、gagent-desktop、mock runner 或其他 Coding Agent 都实现 AgentRunner 接口。
- **Git 发布可控**：默认只本地 branch 和 commit，push 与 PR 需要显式配置。
- **状态必须落库**：每个关键状态变化都写入数据库，支持失败恢复。
- **危险操作默认关闭**：修改敏感文件、直接改 main、自动 push、自动 PR 默认禁止。
- **先闭环再扩展**：MVP 优先跑通本地模拟闭环，不接真实微信和复杂前端。
